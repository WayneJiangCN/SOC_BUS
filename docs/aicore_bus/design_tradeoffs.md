# 设计取舍

## 总体取向

这套模型的定位是：

- 多 AI Core SoC
- NoC-lite
- 事务级
- 面向 CA/ESL 性能趋势分析

因此它主动保留了会影响瓶颈判断的结构，也主动省略了会显著增加复杂度、但对 80% 左右趋势精度贡献不大的细节。

## 保留了什么

### 1. `txn_ctx_`

优点：

- 写请求、写数据、写完成可以可靠配对
- 回包归属清楚
- 目标侧 slot 释放时机清楚

代价：

- 需要一张共享事务表
- 逻辑上比纯直通模型更重

### 2. 端口化 router

优点：

- 能表达方向竞争
- 能分开看输入口和输出口
- 能做每个 output port 的仲裁统计

代价：

- 代码复杂度高于旧版“大 queue router”
- 阅读成本更高

### 3. 端口到端口的 link

优点：

- 包不是瞬移
- 可以表达逐拍发射和延迟后到达
- 可以明确 `src_port -> dst_port`

代价：

- 比单个 `next_hop_time` 更复杂

### 4. target 侧流控

优点：

- 能模拟 OSD、busy、token、credit
- 更容易抓住 target 端真实瓶颈

代价：

- target 路径不再是“发到就算完”

## 明确没做什么

### 1. flit

没做的原因：

- 会显著放大状态数和调度复杂度
- 对当前 SoC 级目标不划算

### 2. VC / credit

没做的原因：

- 当前更关心拓扑、热点、目标瓶颈
- 不是在做 NoC 微结构研究

### 3. 细粒度 crossbar pipeline

没做的原因：

- 当前 output-port RR 已经能表达主要竞争
- 再往下做会让模型更慢、更难校准

## 当前主要抽象

### 1. `REQ / WR_DAT / RSP` 共享一个 output port

含义：

- 当前把不同流量类看成共享同一条方向带宽

优点：

- 简单
- 能直接看到互相压制

缺点：

- 比一些更强的真实硬件更保守

### 2. `rd_rsp_port_num`

含义：

- 用逻辑 lane 数抽象读返回并行能力

优点：

- 容易调参
- 容易做读返回瓶颈敏感性分析

缺点：

- 不是一一对应某个具体物理端口实现

### 3. NIU 仍然是 message-buffer 风格

含义：

- 上游请求先入本地 pending，再注入 ring

优点：

- 接近系统级模型使用方式
- 接口和网络解耦

缺点：

- 比严格端口握手模型更抽象

## 什么时候该继续细化

如果后面出现以下需求，再考虑往更细结构走：

- 要做 input/output port 级利用率校准
- 要对齐更精确的周期数
- 要研究 request/data/response 子网拆分收益
- 要研究更细的读返回物理结构

在这些需求出现之前，当前层级已经适合继续做 workload 驱动的性能分析。
